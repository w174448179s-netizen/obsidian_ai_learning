## 前置知识体系
 
理解该模块需要以下知识储备：
 
### 1. 计算机网络基础
- **HTTP/HTTPS协议**：请求-响应模型、状态码、请求头/响应头
  - 关键点：Authorization头传递API Key，Content-Type协商
- **SSE (Server-Sent Events)**：服务端→客户端单向推送协议
  - 对比：WebSocket（全双工）vs SSE（半双工，浏览器原生支持）
  - 关键点：`text/event-stream` MIME类型、`data:` 字段格式、自动重连机制
- **RESTful API设计**：资源路径设计、请求/响应体 JSON 规范
 
### 2. Java Web 栈
- **Spring Boot**：自动配置、Starter 机制
- **Spring WebFlux**（非阻塞响应式）vs **Spring WebMVC**（传统Servlet）
  - 关键点：流式场景必须选 WebFlux（SSE 需要非阻塞 I/O）
- **WebFilter**：全局过滤器链，用于鉴权、TraceId、日志横切
- **Reactor 响应式编程**：`Mono<T>`（单值）、`Flux<T>`（多值流）、`map/flatMap/doFinally`
 
### 3. 弹性设计模式
- **限流算法**：令牌桶（Token Bucket）、漏桶（Leaky Bucket）
  - 对比：令牌桶允许突发（可累积令牌），漏桶强制匀速
  - Resilience4j RateLimiter 基于令牌桶
- **熔断模式 (Circuit Breaker)**：关闭→开启→半开三种状态
  - 失败阈值计数 → 开启 → 半开试探 → 恢复/继续熔断
- **降级策略编排**：何时触发、如何降级、如何恢复
  - 降级阶梯：主模型→备模型→静默提示
 
### 4. Java 设计模式
- **适配器模式 (Adapter)**：核心模式，将各模型 SDK 差异封装
  - 开闭原则：新增模型只加一个 Adapter 类，不改已有代码
  - 组合优于继承：因第三方 SDK 的请求对象不可继承
- **策略模式 (Strategy)**：不同降级策略可替换、可组合
- **模板方法 (Template Method)**：适配器基类定义 `chat()` 骨架，子类实现具体调用
 
### 5. 并发与异步编程
- **CompletableFuture**：非流式异步调用，编排多个模型调用
  - `supplyAsync`、`thenApply`、`exceptionally`、`orTimeout`
- **响应式流 (Reactive Streams)**：`Publisher` / `Subscriber` 契约
- **线程模型**：WebFlux 使用少量事件循环线程，避免阻塞
 
### 6. 认证鉴权基础
- **API Key 认证**：对称密钥，每次请求携带
  - 痛点：Key 泄露、轮转、多租户隔离
- **JWT (JSON Web Token)**：无状态令牌，可携带用户信息
- **OAuth2 基础**：Client Credentials 流程（机器对机器）
 
### 7. 日志与可观测性
- **MDC (Mapped Diagnostic Context)**：SLF4J 日志上下文，支持 TraceId
  - 关键点：响应式场景需手动传递 MDC（`Hooks.enableAutomaticContextPropagation()`）
- **全链路追踪**：TraceId 贯穿请求全路径
- **健康检查**：探活端点（`/health`），定期检测上游模型可用性
 
### 8. 模型厂商API差异
- **OpenAI 格式**（业界事实标准）：`messages` 数组、`role`/`content`、`stream` 参数
- **Claude (Anthropic)**：不同消息格式、`temperature` 范围 0-1（vs GPT 0-2）
- **阿里云百炼/通义千问**：兼容 OpenAI 格式但有限差异
- **DeepSeek**：基本兼容 OpenAI 但部分参数不同
- **多模态差异**：图片输入格式、Token 计费方式各不相同
 
### 9. 框架工具
- **Spring AI**：官方 AI 框架，提供 ChatClient 封装
  - 详见 [[Spring AI]]
- **Resilience4j**：轻量级限流熔断库
- **Netty**：WebFlux 底层 NIO 框架
 
## 核心实现要点
 
### 架构分层
 
```
[客户端] → WebFilter链（鉴权/TraceId/日志/限流）
         → Router（模型路由 + 灰度）
         → Adapter（协议转换 + 多模型调用）
         → Provider SDK（OpenAI/Claude/百炼...）
```
 
### 适配器体系
 
每个模型一个 Adapter，统一继承自 `ModelAdapter` 接口：
 
```java
public interface ModelAdapter {
    /** 非流式调用 */
    CompletableFuture<ModelResponse> chat(ModelRequest request);
    /** 流式调用，返回响应式流 */
    Flux<String> streamChat(ModelRequest request);
    /** 判断是否支持指定模型类型 */
    boolean supports(ModelType modelType);
    /** 健康检查 */
    boolean healthCheck();
}
```
 
### 参数归一化
 
各模型参数范围不同，在 Adapter 层做双向映射：
 
| 参数 | 前端传入 | GPT 适配 | Claude 适配 | 百炼 适配 |
|------|---------|----------|------------|----------|
| temperature | 0~1 | 0~2 (x2) | 0~1 (x1) | 0~1 (x1) |
| max_tokens | 整数 | max_tokens | max_tokens | max_tokens |
| top_p | 0~1 | 0~1 | 0~1 | 0~1 |
 
### 限流+熔断组合
 
```java
// 标准调用编排
public ModelResponse chat(ModelRequest request) {
    return rateLimiter.executeSupplier(() -> 
        circuitBreaker.executeSupplier(() -> 
            adapter.chat(request).join()
        )
    );
}
```
 
### TraceId 全链路传递
 
```java
@Component
public class TraceIdFilter implements WebFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        // 1. 获取或生成 TraceId
        String traceId = exchange.getRequest().getHeaders()
            .getFirst("X-Trace-Id");
        if (traceId == null) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        // 2. 放入 MDC
        MDC.put("traceId", traceId);
        // 3. 放入响应头
        exchange.getResponse().getHeaders().add("X-Trace-Id", traceId);
        // 4. 传递到下游（响应式上下文）
        return chain.filter(exchange)
            .contextWrite(ctx -> ctx.put("traceId", traceId))
            .doFinally(signalType -> MDC.clear());
    }
}
```

## 架构反思
- **适配器 vs 管道**：接入网关本质是"协议转换+流量控制"的薄层，职责明确。业务逻辑不应渗透进来，否则会退化为"大网关反模式"
- **限流与熔断的粒度**：按模型实例、按用户还是按API Key？粒度越细控制越精确，但维护成本和计算开销也越大
- **异步编程的复杂度**：WebFlux + MDC 组合在响应式线程间传递上下文需要额外处理，Spring Boot 3.2+ 通过 `ObservationRegistry` 简化部分工作
