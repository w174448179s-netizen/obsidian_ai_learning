## 为什么需要接入网关 — 能力驱动拆解
 
### 核心理念
 
> **接入网关是 AI 应用的"门面"。** 它解决一个核心矛盾：前端希望一个统一 API 搞定所有模型，但每个模型厂商的 API 格式、认证方式、速率限制、能力边界完全不同。
 
如果不用网关，代码会变成这样：
 
```java
// 无网关：每个客户端/服务都要处理所有模型差异
if ("gpt-4o".equals(model)) {
    // OpenAI SDK 调用 + 参数范围 0-2
} else if ("claude-sonnet".equals(model)) {
    // Anthropic SDK 调用 + 不同消息格式
} else if ("qwen-max".equals(model)) {
    // 百炼调用 + 兼容OpenAI但有差异
}
// 新增模型？改所有调用方代码
```
 
**网关就是把这种 if-else 爆炸收敛到一层**，对外暴露不变的接口，对内用适配器模式封装变化。
 
### 能力拆解 — 优先级排列
 
| 优先级 | 能力 | 为什么需要它 | 没有它会怎样 |
|--------|------|------------|------------|
| **P0** | 协议转换 | 各模型请求/响应格式不同，前端无法逐一适配 | 前端写死模型绑定，换模型改代码 |
| **P0** | 流式 SSE | LLM 推理耗时数秒，用户不能干等全部结果 | 用户看到白屏数秒，体验不可接受 |
| **P0** | 模型路由 | 多模型并存，前端/业务需要灵活选择 | 硬编码模型名，无法灰度、无法切换 |
| **P1** | 限流 | 模型 API 有价格和 QPS 限制 | API Key 超额扣费，或请求被厂商限流报错 |
| **P1** | 熔断 | 模型可能不可用或响应极慢 | 一个慢模型拖垮整个应用的线程池 |
| **P1** | 认证鉴权 | API Key 不能明文暴露给前端 | Key 泄露 → 盗刷 → 巨额账单 |
| **P2** | TraceId | 请求经网关→过滤链→推理→后置，出问题难以定位 | 分布式排查全靠猜，问题复现靠缘分 |
| **P2** | 降级策略 | 主模型不可用时不能直接抛 500 | 用户被错误页面劝退 |
| **P3** | 健康检查 | 需要主动知道模型是否活着 | 等用户请求超时才被动感知，影响用户体验 |
| **P3** | 超时控制 | 各模型响应速度差异大（GPT-4o 秒回 vs 深度推理模型分钟级） | 统一超时 → 要么常超时，要么等太久 |

## 核心代码
 
### ModelAdapter 接口
 
```java
public interface ModelAdapter {
    /** 非流式 */
    CompletableFuture<ModelResponse> chat(ModelRequest request);
    /** 流式 */
    Flux<String> streamChat(ModelRequest request);
    /** 路由判断 */
    boolean supports(ModelType modelType);
    /** 探活 */
    boolean healthCheck();
}
```
 
### ModelRouter 接口
 
```java
public interface ModelRouter {
    ModelAdapter route(String modelName);            // 精确匹配
    ModelAdapter routeWithGray(ModelRequest req);    // 灰度路由
    void markUnhealthy(ModelType type);              // 熔断反馈
}
```
 
### 限流+熔断编排（伪代码）
 
```java
public Flux<String> chatStream(ModelRequest request) {
    return rateLimiter.defer(ctx ->
        circuitBreaker.defer(ctx2 ->
            modelRouter.route(request.getModel())
                .streamChat(request)
        )
    );
}
```
 
### WebFilter 链顺序
 
```
请求 → TraceIdFilter → AuthFilter → SizeLimitFilter → Controller
```

## 架构反思
 
### 协议转换 vs 大网关反模式
接入网关是"薄层"——只做协议转换和流量控制。**业务逻辑（意图识别、RAG、Agent编排）不能渗透进来**，否则退化为"大网关反模式"：一个模块做所有事，谁也改不动。
 
### 限流熔断的粒度选择
- 按全局：实现简单，但一个暴力用户会饿死所有人
- 按用户：公平，但 Redis 计数器有开销
- 按 API Key：适合 SaaS 多租户，但 Key 数量大了管理复杂
- **起步阶段**：全局 + 按用户双层，后续按需扩展
 
### 流式场景下的复杂性
WebFlux + MDC 在响应式线程间传递上下文是一件需要显式处理的事。Spring Boot 3.2+ 的 `ObservationRegistry` 简化了，但理解其原理（Reactor Context 是一个不可变链表）仍然重要。
 
### "先有后优"原则
Day 1-3 的代码可以接受硬编码配置（如一台开发机上的配置只读），Day 4-6 才引入动态化和弹性。**不要在 Day 1 想清楚 Day 6 的所有边界**，先跑通，再加固。
