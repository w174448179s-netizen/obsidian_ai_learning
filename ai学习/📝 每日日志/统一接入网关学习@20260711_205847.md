# 1. 接入网关 — 统一API，屏蔽模型差异

## 一句话职责
作为AI应用的**统一入口**，对外暴露标准化API，对内屏蔽不同大模型（OpenAI、Claude、通义千问、文心一言等）的差异。

## 核心能力

### 1.1 协议转换
- 将外部统一请求格式，转换为各模型特有的API格式
- 将各模型的差异化响应，转换为统一标准响应

### 1.2 模型路由
- 根据请求参数（模型名、能力要求）路由到对应模型
- 支持灰度发布：部分流量走新模型，部分走旧模型

### 1.3 流量控制
- 限流（Rate Limiting）：防止单个用户过度调用
- 熔断（Circuit Breaker）：某模型不可用时自动降级

### 1.4 认证与鉴权
- API Key校验
- 请求签名验证

## 踩坑记录

### 坑1：流式响应的连接复用
- **问题**：HTTP/1.1长连接在流式场景下容易被客户端关闭
- **解决**：使用WebSocket或Server-Sent Events（SSE），配合心跳保活

### 坑2：模型参数差异
- **问题**：不同模型的temperature范围不同（GPT：0-2，Claude：0-1）
- **解决**：在适配器层做参数归一化，统一接收0-1范围，由适配器转换

### 坑3：Token计费差异
- **问题**：各模型Token计算方式不同，导致计费不准确
- **解决**：统一使用tiktoken库计算Token，或由各适配器自行计算后上报

# 第一轮：协议转换

## 你的理解 ✅
> 基本结构一致，各家特殊参数继承基本结构；响应都是文本结果

**基本正确**，但有个关键细节需要补充：

### 请求转换的完整流程

```
统一请求 → 适配器 → 模型特有请求 → 调用API → 模型特有响应 → 适配器 → 统一响应
```

### 继承 vs 组合

你说的是**继承**思路，但实际更推荐**组合**：

```java
// 不推荐：继承
public class OpenAIRequest extends UnifiedRequest {
    // 加OpenAI特殊字段
}

// 推荐：组合（适配器内部转换）
public class OpenAIAdapter implements ModelAdapter {
    
    // 统一请求 → OpenAI SDK自己的请求对象
    public ChatCompletionRequest convert(ModelRequest request) {
        return ChatCompletionRequest.builder()
            .model(request.getModel())          // 统一字段
            .messages(convertMessages(request)) // 统一字段
            .temperature(request.getTemperature()) // 统一字段
            // OpenAI特殊字段，统一请求里没有
            .frequencyPenalty(0.0)              // 适配器自己填默认值
            .logitBias(null)                    // 适配器自己处理
            .build();
    }
}
```

**为什么不用继承？** 因为各模型SDK的请求对象是第三方库定义的，你没法继承它们。

### 响应转换的关键点

响应不只是"文本结果"，还有这些结构化信息需要提取：

```java
// 统一响应结构
public class ModelResponse {
    private String content;           // 核心：生成的文本
    private int promptTokens;         // 输入Token数（计费用）
    private int completionTokens;     // 输出Token数（计费用）
    private String finishReason;      // stop/length/content_filter
    private Map<String, Object> extras; // 各模型特有信息
}

// OpenAI响应 → 统一响应
public ModelResponse convert(ChatCompletionResult result) {
    ChatCompletionChoice choice = result.getChoices().get(0);
    return ModelResponse.builder()
        .content(choice.getMessage().getContent())
        .promptTokens(result.getUsage().getPromptTokens())
        .completionTokens(result.getUsage().getCompletionTokens())
        .finishReason(choice.getFinishReason())
        .extras(Map.of(
            "system_fingerprint", result.getSystemFingerprint() // OpenAI特有
        ))
        .build();
}
```

# 第二轮：模型路由

## 你的理解 ✅
> 前端没指定就用默认，流量达到阈值就熔断降级

### 路由策略的完整设计

```java
@Component
public class ModelRouter {
    
    private final List<ModelAdapter> adapters; // 所有适配器
    private final ModelConfig config;           // 配置
    
    public ModelAdapter route(String modelName) {
        // 策略1：前端指定
        if (StringUtils.hasText(modelName)) {
            return matchExact(modelName);
        }
        
        // 策略2：未指定，走默认模型
        String defaultModel = config.getDefaultModel();
        return matchExact(defaultModel);
    }
    
    // 灰度路由（进阶）
    public ModelAdapter routeWithGray(ModelRequest request) {
        String userId = request.getUserId();
        
        // 策略3：白名单用户走新模型
        if (config.getGrayUserIds().contains(userId)) {
            return matchExact(config.getGrayModel());
        }
        
        // 策略4：按比例灰度（比如10%流量走新模型）
        if (hashConsistent(userId) < config.getGrayPercent()) {
            return matchExact(config.getGrayModel());
        }
        
        // 策略5：按模型能力路由
        // 如果请求需要"图片理解"，路由到多模态模型
        if (request.requiresVision()) {
            return matchExact("gpt-4o");
        }
        
        return route(request.getModel());
    }
}
```

### 路由 + 熔断的联动

```
请求到达 → 路由选择模型A → 检查模型A状态
    ├── 正常 → 调用模型A
    ├── 熔断开启 → 路由到模型B（降级）
    └── 限流触发 → 返回429或排队
```

## 熔断怎么实现？

### 熔断的三种状态

```
关闭（正常） → 失败率达到阈值 → 开启（拒绝请求）
   ↑                                    ↓
   ←──── 超时后尝试 ←── 半开（试探请求）
```

### Resilience4j 熔断器

```java
// 1. 配置
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)           // 失败率50%触发熔断
    .waitDurationInOpenState(Duration.ofSeconds(10)) // 熔断持续10秒
    .slidingWindowSize(10)              // 统计最近10次请求
    .minimumNumberOfCalls(5)            // 最少5次调用才开始统计
    .build();

CircuitBreaker circuitBreaker = CircuitBreaker.of("openai", config);

// 2. 熔断事件监听
circuitBreaker.getEventPublisher()
    .onStateTransition(event -> 
        log.warn("熔断状态变更: {} → {}", 
            event.getStateTransition().getFromState(),
            event.getStateTransition().getToState()))
    .onFailureRateExceeded(event -> 
        log.error("失败率超过阈值: {}", event.getFailureRate()));

// 3. 在适配器中使用
public ModelResponse chat(ModelRequest request) {
    return circuitBreaker.executeSupplier(() -> {
        return callOpenAI(request);
    });
}
```

# 第四轮：流式处理

## 你的理解 ✅
> 一段一段返回 vs 一次性批量返回

**补充关键点**：

### 流式 vs 非流式的区别

| 特性 | 非流式 | 流式 |
|------|--------|------|
| 响应方式 | 等全部生成后一次性返回 | 生成一点返回一点 |
| 首字延迟 | 高（等全部） | 低（几百毫秒） |
| 用户体验 | 等待转圈 | 打字机效果，体验好 |
| 实现复杂度 | 低 | 高 |
| 适用场景 | 离线处理、批量 | 对话、实时交互 |

### 流式代码实现

```java
// 1. 统一流式接口
public interface StreamAdapter {
    Flux<String> streamChat(ModelRequest request);
}

// 2. OpenAI流式实现（使用WebFlux）
@Component
public class OpenAIStreamAdapter implements StreamAdapter {
    
    private final WebClient webClient;
    
    @Override
    public Flux<String> streamChat(ModelRequest request) {
        return webClient.post()
            .uri("https://api.openai.com/v1/chat/completions")
            .bodyValue(buildStreamRequest(request))
            .retrieve()
            .bodyToFlux(String.class)
            .filter(line -> line.startsWith("data: "))
            .map(line -> line.substring(6)) // 去掉 "data: " 前缀
            .filter(data -> !"[DONE]".equals(data)) // 去掉结束标记
            .map(this::parseToken) // 解析JSON，提取token
            .timeout(Duration.ofSeconds(30)) // 超时控制
            .retryWhen(Retry.backoff(3, Duration.ofSeconds(1))); // 重试
    }
    
    private String parseToken(String data) {
        // 解析 {"choices":[{"delta":{"content":"你好"}}]}
        JsonNode node = objectMapper.readTree(data);
        return node.path("choices").get(0)
                   .path("delta").path("content")
                   .asText("");
    }
}

// 3. 控制器层 - 对外暴露SSE端点
@RestController
public class ChatController {
    
    @PostMapping(value = "/v1/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> streamChat(@RequestBody ChatRequest request) {
        return gateway.streamChat(request)
            .map(token -> ServerSentEvent.builder(token).build())
            .doOnComplete(() -> log.info("流式输出完成"));
    }
}
```

### SSE vs WebSocket

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 服务端→客户端单向 | 双向 |
| 协议 | HTTP | 独立协议 |
| 浏览器支持 | 原生支持 | 原生支持 |
| 重连 | 自动重连 | 需手动实现 |
| 适用场景 | AI流式输出（完美匹配） | 实时协作、游戏 |

**结论：AI流式输出用SSE就够了，不需要WebSocket。**

# 第七轮：认证鉴权 + 全局流水号

## 认证鉴权

```java
@Component
public class AuthFilter implements WebFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String apiKey = exchange.getRequest().getHeaders()
            .getFirst("Authorization");
        
        if (apiKey == null || !isValid(apiKey)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        // 认证通过，把用户信息放入上下文
        String userId = extractUserId(apiKey);
        return chain.filter(exchange)
            .contextWrite(ctx -> ctx.put("userId", userId));
    }
}
```

## 全局流水号（TraceId）

```java
@Component
public class TraceIdFilter implements WebFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        // 1. 从请求头获取（客户端传入）
        String traceId = exchange.getRequest().getHeaders()
            .getFirst("X-Trace-Id");
        
        // 2. 如果没有，自动生成
        if (traceId == null) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        
        // 3. 放入MDC（日志上下文）
        MDC.put("traceId", traceId);
        
        // 4. 放入响应头（回传客户端）
        exchange.getResponse().getHeaders()
            .add("X-Trace-Id", traceId);
        
        return chain.filter(exchange)
            .doFinally(signalType -> MDC.clear());
    }
}
```

### 流水号的作用

```
用户请求 → 网关生成traceId=abc123
    ↓
前置过滤链 → 日志: [traceId=abc123] 输入检查通过
    ↓
语义缓存 → 日志: [traceId=abc123] 缓存未命中
    ↓
意图识别 → 日志: [traceId=abc123] 识别为"闲聊"
    ↓
模型调用 → 日志: [traceId=abc123] 调用GPT-4o耗时2.3s
    ↓
后置处理 → 日志: [traceId=abc123] 脱敏完成
```

**一个traceId贯穿全链路，出问题直接搜这个ID就能看到所有日志。**

# 1. 接入网关 — 统一API，屏蔽模型差异

## 一句话职责
作为AI应用的统一入口，对外暴露标准化API，对内屏蔽不同大模型（OpenAI、Claude、通义千问、文心一言等）的差异。

## 核心能力

### 1.1 协议转换
- 统一请求格式 → 各模型特有请求格式（适配器内部转换）
- 各模型特有响应 → 统一响应格式（提取content、tokens、finishReason等）
- 推荐组合方式而非继承，因各模型SDK请求对象为第三方库定义

### 1.2 模型路由
- 前端指定模型名 → 精确匹配
- 前端未指定 → 使用默认模型
- 灰度路由：白名单用户/按比例灰度
- 按能力路由：如图片理解请求路由到多模态模型
- 路由与熔断联动：模型熔断开启时自动路由到备用模型

### 1.3 流量控制
- **限流**：基于Resilience4j RateLimiter，支持全局限流和按用户限流
- **熔断**：基于Resilience4j CircuitBreaker，三种状态（关闭→开启→半开）
- 限流+熔断组合使用：先限流再熔断最后调用

### 1.4 认证与鉴权
- API Key校验（从Authorization头提取）
- 认证通过后将用户信息放入上下文

### 1.5 全局流水号（TraceId）
- 从请求头获取或自动生成UUID
- 放入MDC日志上下文，贯穿全链路
- 放入响应头回传客户端

### 1.6 流式处理
- 非流式：一次性返回全部结果
- 流式：SSE（Server-Sent Events）逐token推送，打字机效果
- 使用WebFlux Flux<String>实现流式调用
- 支持超时控制、重试机制

### 1.7 降级策略
- 主备切换：主模型失败→备用模型
- 静默降级：返回固定友好提示
- 缓存降级：返回历史相似回答
- 简化模型降级：GPT-4o→GPT-4o-mini→本地小模型
- 功能裁剪降级：完整回答→纯文本→关键词
- 排队降级：削峰填谷，异步处理

### 1.8 辅助能力
- 模型健康检查（探活）：每30秒定时检查，不可用时标记并告警
- 请求/响应日志：记录traceId、路径、耗时、状态码
- 请求大小限制：防止恶意大请求（默认1MB）
- 模型调用超时控制：不同模型设置不同超时时间

## 实现要点

### 适配器模式
- 核心设计模式，将各模型调用差异封装在适配器内部
- 遵循开闭原则：新增模型只需新增Adapter类，不改已有代码
- 无适配器模式会导致if-else分支爆炸

### 限流熔断框架选择
- 推荐Resilience4j：轻量级、纯Java、无外部依赖
- 备选：Sentinel（阿里，功能全）、Spring Cloud Circuit Breaker

### 流式处理协议选择
- 推荐SSE：服务端→客户端单向，基于HTTP，浏览器原生支持，自动重连
- WebSocket：双向通信，适合实时协作场景

### 认证鉴权实现
- 使用WebFilter全局拦截
- 从Authorization头提取API Key
- 验证通过后将用户信息放入ReactiveContext

## 踩坑记录

### 坑1：流式响应的连接复用
- 问题：HTTP/1.1长连接在流式场景下容易被客户端关闭
- 解决：使用SSE配合心跳保活

### 坑2：模型参数差异
- 问题：不同模型的temperature范围不同（GPT：0-2，Claude：0-1）
- 解决：在适配器层做参数归一化，统一接收0-1范围，由适配器转换

### 坑3：Token计费差异
- 问题：各模型Token计算方式不同，导致计费不准确
- 解决：统一使用tiktoken库计算Token，或由各适配器自行计算后上报

### 坑4：模型超时差异
- 问题：不同模型响应速度差异大，统一超时导致频繁超时或等待过久
- 解决：为每个模型配置独立的超时时间
