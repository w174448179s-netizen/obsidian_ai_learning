# 七、RAG检索增强生成

## 一句话职责
通过检索外部知识，增强模型回答的准确性和时效性。

## 文档处理

### 文件上传
- 支持多种文件格式：PDF、DOCX、TXT、MD
- 文件大小限制
- 文件类型校验

### 文档解析
- PDF解析：提取文本内容
- DOCX解析：提取文本和格式
- HTML解析：提取正文内容
- 表格解析：结构化数据提取

### Chunk切分
- 固定长度切分
- 语义切分（基于段落/章节）
- 重叠窗口（相邻Chunk有重叠）
- Chunk大小优化

## Embedding

### 文本向量化
- 使用Embedding模型将文本转为向量
- 向量维度选择
- 模型选择（OpenAI text-embedding、Sentence-BERT等）

### 向量化策略
- 批量向量化
- 增量向量化
- 向量化缓存

## Vector Database

### 支持的向量数据库
- **Milvus**：专业向量数据库，支持大规模数据
- **pgvector**：PostgreSQL向量扩展，适合中小型项目
- **Elasticsearch Vector**：全文检索+向量检索结合
- **Redis Stack**：轻量级，适合快速原型

### 数据库选择考量
- 数据规模
- 查询性能要求
- 部署复杂度
- 成本

## Retrieval

### 相似度搜索
- 计算查询向量与文档向量的相似度
- 常用相似度算法：余弦相似度、欧氏距离

### TopK召回
- 返回相似度最高的K个文档
- K值的选择（影响召回率和准确率）

### 混合搜索
- 关键词搜索 + 向量搜索
- 结果融合

## Rerank

### 二次排序
- 对召回结果进行精排
- 使用交叉编码器模型
- 提升结果相关性

### 排序策略
- 语义相关性排序
- 时间排序
- 重要性排序

## Context Builder

### 拼接Prompt上下文
- 将检索结果拼接为上下文
- 添加来源引用
- 控制上下文长度

### 上下文管理
- 上下文窗口限制
- 重要信息优先
- 冗余信息过滤

## 完整流程

```
用户查询 → 查询向量化 → 向量检索 → TopK召回 → Rerank
            → Context拼接 → Prompt构建 → LLM生成 → 返回结果
```

## 设计考量

- Chunk大小影响检索效果，需要根据文档类型调整
- 向量数据库选择需要考虑数据规模和查询性能
- Rerank可以显著提升检索质量，但会增加延迟
