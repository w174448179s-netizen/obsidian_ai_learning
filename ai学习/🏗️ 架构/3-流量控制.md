# 三、流量控制 Traffic Control

## 一句话职责
通过限流、配额、熔断等手段，保护系统稳定运行。

## 一、Rate Limit（限流）

### 作用
- 控制请求进入系统的速度
- 防止瞬时流量冲垮系统
- 保护 Gateway 和下游服务

### 控制对象
- **QPS（Queries Per Second）**：每秒请求数
- **RPM（Requests Per Minute）**：每分钟请求数
- **TPS（Transactions Per Second）**：可选

### 常用算法
- **Token Bucket（令牌桶）** ★★★★★：最常用，支持突发流量
- **Leaky Bucket（漏桶）**：平滑流量，不支持突发
- **Fixed Window（固定窗口）**：简单但有边界问题
- **Sliding Window（滑动窗口）**：更精确但实现复杂

### Spring Cloud Gateway 实现
- RedisRateLimiter：分布式限流
- RequestRateLimiter Filter：网关层限流
- Redis Lua 保证原子性

### AI平台实践
- 用户级限流（User）
- API Key 限流
- Agent 限流
- 模型限流
- IP 限流（防攻击）

### 特点
- 控制"进入速度"
- 与请求是否结束无关
- 每个 HTTP 请求都会统计一次

## 二、Concurrent Limit（并发限制）

### 作用
- 控制同时执行中的请求数量
- 防止长时间 SSE 占满系统资源
- 保护模型并发能力

### 控制对象
- Running Request
- Running Task
- Active SSE Connection

### 实现方式
- Semaphore（单机）
- Redis Semaphore（分布式）
- Atomic Counter
- Reactive Resource Control

### 生命周期
- HTTP进入 → Acquire Permit → AI生成（SSE输出）→ 请求结束 → Release Permit

### AI平台实践
- 全局并发限制
- 用户并发限制
- Agent 并发限制
- 模型并发限制

### 特点
- 控制"同时执行数量"
- SSE结束才释放
- Client Cancel 必须释放资源

## 三、Quota（配额）

### 作用
- 控制长期资源消耗
- 控制平台成本
- 控制用户套餐额度

### 控制对象
- Token额度
- 调用次数
- 金额预算
- 图片生成次数
- 文件上传额度

### 数据存储
- Redis（实时统计）
- MySQL（持久化）
- 定时同步

### AI平台实践
- 每日 Token
- 每月 Token
- 每日 Chat 次数
- 每月费用预算
- 企业套餐额度

### 特点
- 控制"长期使用量"
- 不会立即恢复
- 按天、按月或按套餐周期恢复

## 四、Circuit Breaker（熔断）

### 作用
- 服务保护
- 故障隔离
- 防止雪崩

### 状态
- **Closed（正常）**：请求正常转发
- **Open（熔断）**：失败率超过阈值，直接熔断
- **Half Open（恢复检测）**：试探性放行少量请求

### 实现
- **Resilience4j**：轻量级，适合响应式
- **Sentinel**：阿里出品，功能丰富

### 配置示例（Resilience4j）

```yaml
resilience4j:
  circuitbreaker:
    instances:
      mainModelCircuitBreaker:
        slidingWindowSize: 20
        failureRateThreshold: 50
        waitDurationInOpenState: 5000
        registerHealthIndicator: true
```

## 五、三者关系

```
AI请求生命周期：
HTTP Request → Rate Limit → Concurrent Acquire → Quota Check
            → AI Service → LLM Streaming（SSE）
            → Concurrent Release → Quota Update（Token统计）
```

| 类型 | 控制目标 | 恢复方式 |
|------|---------|---------|
| Rate Limit | 进入速度 | 下一秒自动恢复 |
| Concurrent Limit | 同时执行数量 | 请求结束释放 |
| Quota | 长期使用量 | 周期重置（天/月） |

## 代码片段

### 限流+熔断编排（伪代码）

```java
public Flux<String> chatStream(ModelRequest request) {
    return rateLimiter.defer(ctx ->
        circuitBreaker.defer(ctx2 ->
            modelRouter.route(request.getModel())
                .streamChat(request)
        )
    );
}
```

## 设计考量

- 限流是"预防"，熔断是"止损"
- 组合编排顺序：rateLimiter → circuitBreaker → modelAdapter
- 起步阶段用全局+按用户双层限流，后续按需扩展
