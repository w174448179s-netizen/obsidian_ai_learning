# 一、接入网关 Gateway

## 一句话职责
统一API入口，处理认证、限流、路由、安全防护等横切关注点。

## Gateway职责

### 请求入口
- 统一API入口
- 协议适配（HTTP/HTTPS）
- 端口监听

### 身份认证
- JWT Token验证
- API Key校验
- OAuth2授权

### 权限校验
- 用户角色校验
- 资源权限校验
- 租户隔离

### 流量控制
- 限流（QPS/并发/配额）
- 熔断（故障隔离）
- 降级（兜底响应）

### 路由转发
- URL匹配
- 服务发现
- 请求转发

### 可观测入口
- TraceId生成
- 日志记录
- 指标采集

### 安全防护入口
- 敏感词过滤
- Prompt Injection检测
- 内容审核

## Gateway技术实现

### Spring Cloud Gateway
- 基于WebFlux的响应式网关
- 非阻塞、高性能
- 支持动态路由

### Reactor Netty
- 响应式网络框架
- 事件循环模型
- 支持海量并发连接

### WebFlux响应式模型
- 非阻塞线程模型
- Flux/Mono响应式流
- 天然支持SSE流式输出

### Filter Chain过滤链
- Global Filter（全局过滤器）
- Route Filter（路由过滤器）
- Ordered排序控制执行顺序

## Filter体系

### Global Filter（全局过滤器）

#### 所有请求执行
- TraceId生成
- 日志记录
- 统一认证
- 全局限流
- 监控指标

#### 执行顺序
- Ordered.HIGHEST_PRECEDENCE：最早执行
- Ordered.LOWEST_PRECEDENCE：最晚执行

### Route Filter（路由过滤器）

#### 指定接口生效
- 某个Agent限制
- 某个模型限制
- 特殊业务规则

#### 常用过滤器
- RequestRateLimiter：限流
- CircuitBreaker：熔断
- AddRequestHeader：添加请求头
- RemoveRequestHeader：移除请求头

### Filter执行顺序

#### Ordered排序
- 数值越小，优先级越高
- 全局过滤器和路由过滤器统一排序

#### 优先级控制
- 认证过滤器优先执行
- 限流过滤器次之
- 路由过滤器最后

#### 前置过滤
- 认证、限流、安全检查

#### 后置过滤
- 日志记录、指标采集、响应处理

## 网关核心规则

- ❌ 禁止全局JSON参数改写Filter，保证主线SSE纯透传
- ✅ 2条核心路由：主线同协议路由 + 独立备用异构路由
- ✅ Header约定：X-Logic-Model-Id 标记模型ID
- ✅ 弹性策略：Resilience4j熔断、Redis分布式令牌桶限流
- ✅ 校验逻辑：JWT验签、JWKS公钥缓存、SSO身份校验

## 代码片段

### WebFilter链顺序

```
请求 → TraceIdFilter → AuthFilter → SizeLimitFilter → Controller
```

### 网关配置示例

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: main-openai-route
          uri: lb://ai-agent-service
          predicates:
            - Path=/v1/chat/completions
            - Header=X-Logic-Model-Id, qwen-openai|deepseek-openai
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
            - name: CircuitBreaker
              args:
                name: mainModelCircuitBreaker
                fallbackUri: forward:/fallback/same-protocol-backup
```

## 设计考量

- 网关是"薄层"，只做协议转换和流量控制
- 业务逻辑不能渗透进来，否则退化为"大网关反模式"
- 流式场景下需要注意线程模型，避免阻塞EventLoop
