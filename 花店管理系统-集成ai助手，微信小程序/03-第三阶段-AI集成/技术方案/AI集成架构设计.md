## 关键设计决策

### 1. 后端代理模式

小程序不直接调用大模型API，统一由后端代理。原因：

- API Key安全（不暴露在前端）
- 统一做内容安全审核
- 统一做频率控制和计费
- 支持流式输出（后端WebSocket/SSE → 小程序）

### 2. RAG + Function Calling 双引擎

- **RAG**用于知识密集型场景（客服、花语、养护）
- **Function Calling**用于需要实时数据的场景（库存查询、订单查询、花束推荐）
- 两种模式可根据用户问题自动切换

### 3. 内容安全

- 用户输入 → 后端文本安全检测 → 通过后继续 → 不通过返回安全提示
- AI输出 → 后端二次校验 → 通过后返回前端 → 不通过返回兜底话术
- 对接微信内容安全API + 自定义敏感词库

### 4. 会话管理

- 后端维护对话会话（session_id + 消息历史）
- 单次会话保留最近N轮对话作为上下文
- 超时自动清理（如30分钟无操作）
- 不同场景（客服/花语/推荐）使用不同的system prompt

### 5. 流式输出

- 后端接收大模型流式响应
- 通过WebSocket实时推送给小程序前端
- 前端逐字渲染，提升用户体验
- 网络异常时后端重试或降级为非流式返回

## 技术栈

| 组件 | 推荐方案 | 说明 |
|------|----------|------|
| 大模型API | DeepSeek V3 / Qwen | 主力模型，性价比高 |
| 备用模型 | 豆包2.0 | 降级或简单场景使用 |
| 向量数据库 | ChromaDB | 开源免费，适合中小规模 |
| Embedding模型 | DeepSeek Embedding / 通义Embedding | 文本向量化 |
| 对话框架 | 自研轻量Agent | 不引入重框架，保持灵活 |
| 流式通信 | WebSocket | 后端→前端实时推送 |
| 内容安全 | 微信msgSecCheck + 自定义词库 | 双重保障 |
